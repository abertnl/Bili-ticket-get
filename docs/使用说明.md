# B 站会员购抢票工具使用说明

本文面向实际使用者，按“安装 → 登录 → 配置 → 开抢 → 处理异常”的顺序说明如何使用本地 Web 版会员购抢票工具。

> 仅供个人买票自用。请遵守 B 站用户协议，合理设置请求间隔，不要用于黄牛囤票、批量刷单或其他滥用场景。

## 1. 环境准备

### 1.1 安装 uv

本项目使用 `uv` 管理 Python 依赖。确认本机已安装 `uv`：

```bash
uv --version
```

如果没有安装，可参考 uv 官方文档安装：<https://docs.astral.sh/uv/>

### 1.2 安装项目依赖

进入项目目录：

```bash
cd Bili-ticket-get
```

安装依赖：

```bash
uv sync
```

## 2. 启动工具

在项目根目录执行：

```bash
uv run main.py
```

启动成功后，浏览器打开：

```text
http://127.0.0.1:8000
```

页面包含三个主要区域：

- **登录**：扫码登录或 Cookie 登录。
- **配置**：选择演出、场次、票档、购票人和抢票参数。
- **抢票监控**：启动/停止抢票，查看实时日志。

## 3. 登录账号

推荐优先使用扫码登录。

### 3.1 扫码登录

1. 打开本地页面 `http://127.0.0.1:8000`。
2. 进入「登录」标签页。
3. 点击「生成二维码」。
4. 使用 B 站手机客户端扫码并确认登录。
5. 页面右上角显示“已登录”即表示成功。

### 3.2 Cookie 登录

如果扫码不方便，可以使用 Cookie 登录。

1. 从浏览器中提取 `show.bilibili.com` 的 Cookie。
2. 确保 Cookie 中包含：
   - `SESSDATA`
   - `bili_jct`
   - `DedeUserID`
3. 粘贴到「Cookie 登录」输入框。
4. 点击「使用 Cookie 登录」。

详细提取方法见：[获取Cookie.md](获取Cookie.md)

> Cookie 等同账号登录凭证，不要发给别人，也不要提交到 Git。工具会把登录信息写入本地 `config.json`，该文件已被 `.gitignore` 忽略。

## 4. 配置抢票信息

进入「配置」标签页。

### 4.1 加载演出

在「演出链接或 project_id」中填入任意一种：

- 会员购演出链接
- 直接填写 `project_id`

然后点击「加载」。

加载成功后，页面会显示演出名称，并填充：

- 场次下拉框
- 票档下拉框

### 4.2 选择场次与票档

1. 在「场次」下拉框选择目标场次。
2. 在「票档」下拉框选择目标票价/票档。

票档后面会显示价格和售卖状态提示。

### 4.3 加载购票人

点击「加载购票人」。

如果账号已保存购票人，页面会列出可选购票人。勾选本次要使用的购票人。

如果没有购票人，请先到 B 站 App 或会员购页面添加实名购票人，再回到工具重新加载。

### 4.4 设置抢票参数

常用参数说明：

| 参数 | 说明 | 建议 |
| --- | --- | --- |
| 数量 | 本次购买张数 | 与购票人数保持一致 |
| 开抢时间 | 到点后自动开始 | 留空表示立即开始 |
| 请求间隔(ms) | 两次请求之间的等待时间 | 默认 `800`，风控频繁时调大 |
| 最大尝试次数 | 达到次数后自动停止 | 默认 `300` |
| 预热提前量(s) | 开抢前提前刷新凭据、加载购票人和票价 | 默认 `30` |
| 限流退避(ms) | 遇到 `429` / `412` 时的最小等待 | 默认 `2000`，频繁限流时调大 |
| 网络退避上限(ms) | 网络异常指数退避的最大等待 | 默认 `3000` |
| 启用回流票监控 | 售罄期间先低频查票档，发现回流后自动下单 | 适合等待退票/释放库存 |
| 监控间隔(ms) | 回流监控轮询间隔 | 默认 `5000`，不要低于 `1000` |
| 监控截止时间 | 没抢到时最晚停止监控的时间 | 启用回流监控时必填 |
| 验证码方式 | 风控时如何处理极验 | 默认 `manual` |
| rrocr token | 第三方打码 token | 仅 `rrocr` 模式需要 |

填写完成后点击「保存配置」。

## 5. 开始抢票

进入「抢票监控」标签页。

1. 确认配置已保存。
2. 点击「开始抢票」。
3. 观察实时日志。
4. 如需中断，点击「停止」。

如果启用了回流票监控，工具会在开抢时间后先按「监控间隔」检查目标票档状态；发现疑似可售后进入下单冲刺。下单时再次返回售罄会回到监控，直到抢中、达到监控截止时间、达到最大尝试次数或手动停止。

监控页会显示：

- 当前尝试次数
- 最近返回码
- 当前状态
- 最近耗时、平均耗时
- 网络异常次数
- 下次重试原因和等待时间
- 实时日志

订单生成后，或接口提示已有待支付订单时，页面会提示成功，并通过已配置的手机提醒渠道推送通知。你需要尽快回到 B 站完成支付，通常支付窗口只有约 10 分钟。

## 6. 风控与验证码

当接口返回 `-352` 时，表示触发风控，需要人机验证。

### 6.1 manual 模式

默认模式是 `manual`。

触发风控后，页面会弹出验证码窗口，显示：

- `gt`
- `challenge`

你需要完成极验验证，并把验证结果回填：

- `validate`
- `seccode`

提交后，工具会继续抢票。

### 6.2 rrocr 模式

如果配置了 `captcha_mode = rrocr` 并填写 `rrocr_token`，工具会尝试调用第三方打码服务。

使用第三方服务前请自行评估：

- 账号安全风险
- 服务可用性
- 合规性
- 成本

本项目只提供可插拔接口，不内置验证码破解逻辑。

## 7. 推送通知

当前支持三种待支付提醒：

- Bark
- Server酱
- macOS 信息 App（适合 Mac mini 通过 iMessage 发到手机）

### 7.1 Bark

填写 Bark URL，例如：

```text
https://api.day.app/your_key
```

订单生成或已有待支付订单时会发送：

```text
会员购订单待支付
订单 xxx 已生成，请在 10 分钟内支付
```

### 7.2 Server酱

填写 Server酱 Key，例如：

```text
SCTxxxxxxxx
```

留空则不发送该渠道。

### 7.3 macOS 信息 App

如果工具运行在 Mac mini 上，并且这台 Mac 已登录 Apple ID、信息 App 能正常发送 iMessage，可以填写「iMessage 收件人」：

```text
+8613800000000
```

也可以填写对方 Apple ID 邮箱。订单生成或已有待支付订单时，工具会调用系统 `osascript` 通过信息 App 发送提醒。首次使用时 macOS 可能会弹出自动化权限确认，需要允许终端或运行工具的应用控制“信息”。

## 8. 配置文件

配置保存在项目根目录：

```text
config.json
```

首次保存配置或登录后会自动生成。

示例结构见：

```text
config.example.json
```

常见字段：

| 字段 | 说明 |
| --- | --- |
| `cookie` | 登录 Cookie |
| `project_id` | 演出 ID |
| `screen_id` | 场次 ID |
| `sku_id` | 票档 ID |
| `buyer_ids` | 购票人 ID 列表 |
| `count` | 购买数量 |
| `start_time` | 开抢时间，ISO 格式 |
| `interval_ms` | 请求间隔，单位毫秒 |
| `max_attempts` | 最大尝试次数 |
| `prewarm_seconds` | 开抢前预热提前量，单位秒 |
| `rate_limit_backoff_ms` | 限流/拦截响应的最小退避，单位毫秒 |
| `network_backoff_max_ms` | 网络异常退避上限，单位毫秒 |
| `return_monitor_enabled` | 是否启用回流票监控 |
| `monitor_interval_ms` | 回流票监控间隔，单位毫秒 |
| `monitor_end_time` | 回流票监控截止时间，ISO 格式 |
| `captcha_mode` | 验证码处理方式 |
| `notify` | 推送配置，包含 Bark、Server酱、iMessage 收件人 |

不要把 `config.json` 发给别人，因为里面可能包含 Cookie。

## 9. 常见返回码

| 返回码 | 含义 | 处理建议 |
| --- | --- | --- |
| `0` | 成功 | 立即去 B 站支付 |
| `429` | 请求过于频繁 | 增大请求间隔或限流退避 |
| `9000xx` | 会员购常见繁忙类错误 | 通常可继续重试 |
| `-352` | 风控校验失败 | 完成人机验证 |
| `412` | 风控较重 / 请求被拦截 | 降低频率，必要时更换网络 |
| `100009` | 库存不足 / 售罄 | 可继续等待回流票 |
| `100048` | 已经下单，请勿重复下单 | 停止抢票，去订单页查看 |

## 10. 使用建议

- 开抢前提前登录，确认账号状态正常。
- 提前加载演出、场次、票档和购票人，避免临近开抢时手忙脚乱。
- `interval_ms` 不要设置过低。默认 `800ms` 已比较激进，频繁 429 或风控时建议调到 `1000ms` 以上。
- `rate_limit_backoff_ms` 控制 429/412 后的保守等待，频繁限流时建议调到 `3000ms` 以上。
- `prewarm_seconds` 用于把购票人、票价和凭据刷新前移到开抢前，通常保持默认即可。
- 回流票监控的 `monitor_interval_ms` 建议保持 `5000ms` 或更高，避免长时间高频刷新演出接口。
- 同一账号不要同时在多个脚本或设备上抢同一个项目。
- 抢中后不要关闭页面，先去 B 站完成支付。

## 11. 故障排查

### 页面打不开

确认服务是否启动：

```bash
uv run main.py
```

确认浏览器访问的是：

```text
http://127.0.0.1:8000
```

### Cookie 登录失败

检查 Cookie 是否包含：

- `SESSDATA`
- `bili_jct`
- `DedeUserID`

也可以直接改用扫码登录。

### 加载购票人失败

可能原因：

- 未登录或 Cookie 失效
- 账号没有保存购票人
- 会员购接口变更

先重新登录，再到 B 站 App 添加购票人后重试。

### 一直返回 429

说明请求过于频繁。增大 `interval_ms`，例如改为 `1000`、`1500` 或更高。

### 触发 -352 风控

按页面提示完成人机验证；如果频繁触发，适当降低频率，不要多开。
